# AI Dynamic Pricing Dashboard (Streamlit MVP)

Приложение для динамического ценообразования: загрузка транзакций, прогноз спроса, подбор цены, what-if сценарии и экспорт отчёта.

## Установка
```bash
pip install -r requirements.txt
```

## Запуск
```bash
streamlit run app.py
```

## Формат входных данных
### Минимально обязательные поля
- `date`
- `product_id`
- `price`

### Рекомендуемые поля
- `quantity`, `revenue`, `cost`, `discount`, `freight_value`, `stock`, `rating`, `category`

Если `quantity/revenue/cost` отсутствуют, используются fallback-правила:
- `quantity = 1`
- `revenue = price * quantity`
- `cost = 0.65 * price` (proxy)

См. пример файла: `sample_transactions.csv`.

## Валидация входа
Приложение:
- удаляет строки с невалидными датами;
- удаляет строки с невалидной/неположительной ценой;
- сообщает о пропусках, дубликатах, короткой истории;
- даёт понятную ошибку при отсутствии обязательных полей или пустом наборе.

## Метрики и логика расчёта
Используются разные сущности, их нельзя смешивать:
- **raw profit** — прямой расчёт прибыли без штрафов за риск;
- **adjusted profit** — raw profit минус штрафы за uncertainty/disagreement и рисковые шаги;
- **baseline** — прогноз на текущей цене;
- **scenario/what-if** — прогноз при ручных параметрах;
- **actual/predicted** — факт истории и модельный прогноз.

Оптимизация ведётся по режимам:
- `maximize_profit`
- `maximize_revenue`
- `protect_volume`
- `balanced_mode`

## Как читать графики
- **Кривая прибыли**: X = цена-кандидат, Y = adjusted profit; вертикали показывают текущую/рекомендуемую цену.
- **Scenario comparison**: показываются profit/revenue/sales по каждому сценарию.
- **Sensitivity heatmap**: оси `price` и `demand_multiplier`, цвет = `profit`.
- **What-if vs baseline**: сравнение дневного спроса и прибыли относительно baseline.

## Confidence и ограничения
- Confidence рассчитывается эвристически и не является гарантией.
- При короткой истории (<60 дней) и/или высоких ошибках рекомендация должна трактоваться консервативно.
- При малом объёме данных избегайте агрессивных изменений цены.

## Ограничения модели
- `cost` может быть proxy, если отсутствует факт.
- Модель чувствительна к качеству исторических данных и стабильности ценовых изменений.
- Экстраполяция за пределы train-диапазона цен ограничивается clipping.

## Тесты
```bash
pytest -q
```

Покрыты проверки: нормализация, ошибки входа, регрессия против leakage через backward fill, сценарные дельты, sensitivity grid.

## Структура модулей (что делает каждый файл)

### Точка входа UI
- `app.py`  
  Streamlit-интерфейс: загрузка файлов, элементы управления, визуализации, запуск сценариев и отображение результатов.
  В этом файле не должна жить тяжёлая расчётная логика модели.

### Расчётный слой
- `app_analysis_runner.py`  
  Оркестрация расчёта по входному контексту UI (`run_analysis_from_context`):
  - выбор ветки Universal/Legacy;
  - чтение и подготовка CSV;
  - применение маппинга колонок;
  - вызов `run_full_pricing_analysis*`.

- `pricing_core/core.py`  
  Ядро модели и алгоритмов: подготовка фичей, обучение/прогноз, оптимизация цены, what-if проекции, вычисление метрик.

- `calc_engine.py`  
  Вспомогательные вычислительные процедуры для базовых MVP-расчётов и совместимости тестов.

### Домен/сценарии
- `app_domain.py`  
  Доменные заготовки для сценариев (например, дефолтный набор Baseline/Scenario A/B/C и стартовые параметры).

- `what_if.py`  
  Пакетный прогон сценариев и построение сетки чувствительности (`run_scenario_set`, `build_sensitivity_grid`).

### Presentation-слой (подготовка данных для UI)
- `app_presenters.py`  
  Подготовка данных к показу пользователю:
  - человеко-читаемые тексты решения;
  - таблица отличий параметров сценариев;
  - форматирование значений факторов.
  Здесь допускается форматирование для таблиц/текста, но не обучение модели.

- `recommendation.py`  
  Генерация бизнес-нарратива и пояснений для рекомендуемого действия.

### Входные данные и схема
- `data_adapter.py`  
  Универсальная нормализация транзакций: приведение колонок, типов и fallback-правил.

- `data_schema.py`  
  Каноническое описание полей и требований для сопоставления входных CSV.

### Тесты
- `test_smoke_mvp.py` — end-to-end smoke и регрессии по основным пользовательским потокам.
- `test_calc_engine.py` — модульные проверки вычислительного слоя.
- `test_refactor_pricing_core.py` — проверки консистентности после рефакторинга pricing core.
